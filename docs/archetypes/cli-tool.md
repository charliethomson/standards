# Archetype: CLI / service tool

A standalone binary distributed to machines (not a hosted service with clients). It may be
its own repo or a `bin/` member inside a product monorepo. The distinguishing feature is
**distribution through the registry with an install manifest**.

Reference: a `someproduct` CLI and a `courier` background-uploader binary.

## Repo shape

```
<tool>/  (or bin/<tool>/ inside a product)
├── Cargo.toml
├── build.rs                version derivation (+ optional libbuildinfo emit)
├── src/main.rs             clap entrypoint; product_name!("dev.thmsn.<product>.cli")
├── completions/            shipped shell completions (or generated by a subcommand)
├── .github/workflows/      cli.release.yml → publish to registry (see ci-cd.md naming)
└── standards/              this submodule (or inherited from the parent monorepo)
```

## Standards that apply

| Standard | How it applies here |
|---|---|
| [Versioning](../versioning.md) | Derived `MAJOR.MINOR.<count>` via `build.rs`; wired into clap `#[command(version = env!("APP_VERSION"))]`. |
| [Build info](../build-info.md) | `libbuildinfo` recommended — a `--version`/`version` subcommand that prints commit hash + dirty flag is invaluable for a distributed binary. |
| [Identifiers](../identifiers.md) | `dev.thmsn.<product>.cli` (or `.courier`, etc.) via `libproduct`. |
| [Registry publishing](../registry-publishing.md) | **The defining standard** — published as registry `kind = cli`; see below. |
| [CI/CD](../ci-cd.md) | `cli.release.yml` builds per-target artifacts on self-hosted runners and publishes. |
| [Testing](../testing.md) | Test command/arg logic and parsing; completions generation. |
| [Workflow](../workflow.md) | Commit to `main`. |
| [Configuration](../configuration.md) | `libconfig` (`Loader` / `config!{}`); defaults→TOML→shared→prefixed. |
| [Error handling](../error-handling.md) | `thiserror`/`liberror` trinity; map to a clean exit code/message. |
| [Rust conventions](../rust-conventions.md) | Edition 2024, clippy pedantic, git deps. |
| [`lib*` ecosystem](../lib-ecosystem.md) | Reuse `libsignal` (shutdown), `libcmd`/`libwhich`, etc. |
| Service architecture / Data / Contracts / Auth / Web | **N/A** (a CLI is not a hosted service). |
| Branding / Deployment / Platform UX | **N/A** (a CLI ships via the registry, not a deploy). |

## Registry distribution

Published with `kind = cli`, which supports **multi-target artifacts** (one per
os/arch/libc) under a single version, plus an **install manifest** describing how to set it
up on a machine:

```json
{
  "bin_name": "<tool>",
  "version_command": ["<tool>", "--version"],
  "symlinks": { "<alias>": "<tool>" },
  "man": ["man/<tool>.1"],
  "completions": {
    "bash": { "generate": ["<tool>", "completions", "bash"] },
    "zsh":  { "generate": ["<tool>", "completions", "zsh"] },
    "fish": { "bundled": "completions/<tool>.fish" }
  }
}
```

Install is via the registry's platform-detecting installer (`GET /install/<slug>` for
Unix, `…/install/<slug>.ps1` for Windows), which selects the right artifact, verifies the
SHA-256, and applies the manifest. Publish per release with
`charliethomson/registry/.github/actions/publish@main` (or `scripts/registry-publish.sh`).
See [registry-publishing.md](../registry-publishing.md).

## Checklist

- [ ] `build.rs` derives the version; `--version` reports it (plus commit hash via `libbuildinfo`).
- [ ] `product_name!("dev.thmsn.<product>.<tool>")` set.
- [ ] Release workflow builds per-target artifacts on self-hosted runners.
- [ ] Published to the registry as `kind = cli` with a complete install manifest.
- [ ] Shell completions shipped or generated by a subcommand.
- [ ] Command/parsing logic tested.
